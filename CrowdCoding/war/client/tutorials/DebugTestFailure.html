Debug a test failure

In Debug a Test Failure, your goal is to find and correct any bug(s) that caused a function to fail one of its unit tests.

CrowdCode provides a test runner listing the result of the test to debug ( failing test panel )

and the results of the previous passing functionâ€™s tests, if any ( other tests panel ).
Failing tests are marked with red and passing tests with green

Your goal is to use the provided tools to make the failing test pass without letting the other tests fail!

Each test result lists the inputs and outputs for the test, as well as information about errors (if any).

For failed tests, the default view shows you the diff between the expected and the actual return value.

Green background is the the expected field value and red background is the actual field value. The fields without a background color, have the same value.

You can always switch to the normal mode.

The function code editor is the main tool of the CrowdCode debugger.

If during the execution of the current test, one exception is raised, you can see the details hovering the mouse on the x icon.

You can monitor a value during the text execution using the console.log(...) function. When you use console.log(...) and re-run the tests, values are shown passing with the mouse by the i icon.

What happens if the bug is not in the function itself but is in a function that it calls? All the functions called during the current test execution, are highlighted in the code editor window.

By clicking on the highlighted functions, you can see the stubs popup for that callee. It shows you all the sets of input with whom the function has been called and, for each set of input, you are allowed to edit the return value.

For example, here a function sum is being called with the parameters 100 and 40 and is returning the output of 142. This does not seem to match its specified behavior. To solve this problem, you can edit the return value, replacing 142 with 140. This creates a stub.
When you re-run the tests, CrowdCode will use the return value specified by the stub rather than the actual return value, executing the function call as if it had returned 140. After submitting the Debug microtask, each stub will be translated into a test for the called function, expressing the desired behavior.

After doing changes to the function code or to the stubs, re-run the tests to see the changes

It might also be that the function itself is correct and the test data is wrong. In this case, you should report an issue with the test.

Reported tests are marked with orange.

Remember: you can submit the microtask only if all the tests are passing (or you disputed the failing tests).

If you think that the function is far to be complete, sketch some behavior with pseudocode or function stubs and submit it ( In case of pseudocode or function stubs present in the function code, you can submit with failing tests too).
